# セキュリティ

> 本プロジェクト固有の入力検証・認証契約・RLS ポリシーは `docs/06-security-specification.md` を正とする。本ファイルはスタック横断の共通方針。

## 認証・認可

- 認証方式はプロジェクト要件に応じて選択する（本プロジェクトは Supabase Auth / Google OAuth）。
- API エンドポイントごとにアクセス制御を設定する（公開 / 認証必須 / ロールベース）。
  - 本プロジェクト: 閲覧系 GET は未認証可、更新系（POST/PATCH）は `Authorization: Bearer <token>` 必須で未認証は `401`。

## 通信・アクセス制御

- 全通信は **HTTPS** を必須とする。
- **CORS** は許可するオリジンを明示的に指定する（`*` は本番で使用しない）。
- **レートリミット** を導入する（認証系エンドポイントは特に厳しく制限）。

## インジェクション対策

- **SQL インジェクション**: ORM またはパラメータバインドを必須とする。生 SQL の文字列結合は禁止。
- **XSS**: ユーザー入力のサニタイズ + CSP ヘッダー設定 + 出力エスケープの多層防御（`dangerouslySetInnerHTML` を使用しない）。
- **CSRF**: セッション認証の場合は CSRF トークン必須。Bearer Token 認証（Cookie に保存しない場合）は不要。

## シークレット管理

- ローカル環境: `front/.env.local`（`.gitignore` に追加必須）
- 本番環境: Secret Manager 等のマネージドサービスを使用（`SUPABASE_SERVICE_ROLE_KEY` はサーバー用途のみ・クライアントに露出しない）
